## Regression caught by the LOCAL one-test loop (not CI)
`ATS_ECR_01` passed on 1.38 but FAILED on 1.48: after login the app sat on a pushed
**"Work Orders"** screen (`assetsTab=false, sitePicker=false`). Root cause chain:
1. `LoginPage.handleSavePasswordTurbo()` detected a post-login popup, the popup
   auto-dismissed before the targeted dismissers ran, and the **blind coordinate tap
   (140,600) fallback** fired on the live Dashboard вАФ
2. вАФ where it landed on the v1.48 **"No Active Work Order вАФ Tap to select" card**,
   pushing the Work Orders screen (nav title `Work Orders`, `BackButton`,
   `Start New Work OrderвА¶`), which hides the tab bar and breaks all navigation.

## Fixes (validated on a fresh 1.48 install, full login path)
1. **Guard the blind tap** вАФ new `LoginPage.appScreenIsInteractable()`: if any known
   app screen (Dashboard / Site picker / Welcome / Login markers) is visible, the
   popup is gone and the coordinate tap is SKIPPED. Log proof:
   `вЪ° Popup gone and app screen interactable вАФ skipping blind coordinate tap`.
2. **Heal in AssetPage.recoverSiteContext** вАФ step 0 `backOutOfWorkOrdersScreen()`:
   detects `Start New Work Order`/`Available Work Orders` + `BackButton`, taps Back
   (accessibilityId, label fallback), re-checks the Assets tab.
3. **Heal in SiteSelectionPage.waitForDashboardReady** вАФ while polling, if the
   Work Orders screen is detected, tap `BackButton` and keep waiting; login now ends
   on the Dashboard for every suite.

Local validation: `ATS_ECR_01` fresh-install run PASSED (2m01s) with the guard firing;
warm run PASSED (11s). `mvn -o test-compile` clean; 26/26 driver-free self-tests green.
